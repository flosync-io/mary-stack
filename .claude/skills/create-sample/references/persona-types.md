# Persona Types — Fixed Fields and Plain-English Descriptions

This file defines the four operator persona types used in Alex evaluation fixtures.
Fixed fields are NEVER generated by the LLM — copy them verbatim.
LLM-generated fields are listed separately.

---

## INTAKE MAPPING — How to translate user descriptions to persona_type

When a user describes an operator during the ReAct intake conversation, map their description:

| What the user says | persona_type |
|---|---|
| "new to CNC", "doesn't know the machine", "first few months", "doesn't know terms", "would need guidance to find things", "new hire", "learning", "confused by terminology" | naive_operator |
| "experienced", "been doing this years", "already tried something", "knows the machine", "uses correct terms", "multi-year", "veteran-ish", "knows what they're doing" | experienced_operator |
| "thinks they know what's wrong", "already diagnosed it", "confident", "pushes to skip steps", "says they know the cause", "sure it's X", "leading question" | leading_operator |
| "being vague", "pushes back on escalation", "evasive", "tries rephrasing", "won't accept Mary's answer", "adversarial", "won't cooperate" | adversarial_operator |

If a user's description is ambiguous, pick the closest match and confirm with them in plain English before generating.

## 1. naive_operator

**Plain English for intake conversation:**
"New to CNC — only been on the floor a few months. Doesn't know machine terminology or component names. Describes things by how they look or sound ('the little window thing', 'that handheld monocular thing'). Needs Mary to guide them to where things are before they can do a check. Won't understand technical terms unless they're explained."

**Fixed fields (copy verbatim — never LLM-generated):**
```yaml
type: naive_operator
name: Sam
experience_months: 2           # use a value 1–3
skill_level: novice
patience:
  initial_patience: 7
  decay_per_turn: 0.4
  quit_threshold: 2
  quit_behavior: frustrated_disconnect
```

**LLM generates for this type:**
- `vocabulary_uses` — everyday floor language, no technical terms; seeded from FM's `operator_phrases`
- `vocabulary_avoids` — ALL KB technical terms for this FM (use `jargon_to_avoid_for_naive` from schema-guide.md)
- `behavior_traits` — 4–6 bullets; must include: asks where things are, describes by appearance, unfamiliar with alarm codes
- `frustration_triggers` — specific to THIS FM's check wording (see anti-patterns.md §7.4)
- `description` — paragraph describing Sam in plain terms

**Key constraint:** `requires_guidance_to_find: true` must be set in the context for any check that involves a named component Sam wouldn't know by name. See schema-guide.md for each check's `requires_guidance_for_naive` flag.

**LLM generates for this type:**
- `vocabulary_uses` — correct floor jargon, alarm codes, component names; seeded from FM's check wording and operator_phrases
- `vocabulary_avoids` — SINUMERIK parameter numbers (will ask Mary to translate to plain terms if used)
- `behavior_traits` — 4–6 bullets; must include: states alarm code upfront, already tried something, dislikes redundant questions
- `frustration_triggers` — specific to THIS FM; must reference concrete observable behaviors not vague tone judgments (see anti-patterns.md §7.4)
- `description` — paragraph describing Pete

## 3. leading_operator

**Plain English for intake conversation:**
"Operator who comes in already convinced they know what's wrong. States a diagnosis upfront and pushes to skip the verification checks ('I know it's X, just tell me how to fix it'). Their self-diagnosis may be WRONG — that's the test. Mary must still run the checks even when the operator is confident."

**Fixed fields (copy verbatim — never LLM-generated):**
```yaml
type: leading_operator
name: Riya
experience_months: 24          # use a value 18–36
skill_level: intermediate
patience:
  initial_patience: 6
  decay_per_turn: 0.6
  quit_threshold: 2
  quit_behavior: angry_escalation
```

**LLM generates for this type:**
- `vocabulary_uses` — mix of correct terms and some floor language; knows enough to sound confident
- `vocabulary_avoids` — SINUMERIK parameter numbers; deep programming terms
- `behavior_traits` — 4–6 bullets; must include: states confident diagnosis, pushes to skip checks, may be impatient when Mary doesn't accept their diagnosis immediately
- `frustration_triggers` — specific to THIS FM's check wording
- `description` — paragraph describing Riya

**CRITICAL for fixtures:** The leading operator's self-diagnosis in the opening complaint should be plausible but WRONG (or at least unconfirmed). This tests CauseSkipDetection — Mary must run checks even when Riya sounds certain.

## PATIENCE SYSTEM — How it works (for reference when writing frustration_triggers)

```
patience_level = initial_patience
per Mary turn:
  patience_level -= decay_per_turn
  for each trigger that fires:
    patience_level -= 1.5  (hardcoded FRUSTRATION_PENALTY)
  if patience_level <= quit_threshold:
    Alex quits the conversation
```

Math check: naive operator with initial=7, decay=0.4, quit=2 → (7-2)/0.4 = 12.5 free turns before quit (no triggers firing). With triggers firing every 2 turns, that's roughly 7–8 turns. Calibrate frustration_triggers so they fire on genuine provocations, not every turn.

**Trigger writing rules (from anti-patterns.md):**
- SPECIFIC word lists beat "jargon the operator wouldn't know"
- Sentence counts beat "long explanation"
- Observable actions beat "tone" or "condescending"
- Reference what was already said in the conversation

**WRONG trigger:** `"Mary uses technical jargon Sam wouldn't understand"`
**RIGHT trigger:** `"Mary uses one or more of these exact words without explaining them: refractometer, Brix, PDS band, tramp oil, emulsion"`
