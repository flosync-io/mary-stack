## Decisions

**Status vocab renamed (2026-06-25):** green‚Üíresolved, amber/red‚Üíunresolved. Driven by contracts/runtime.schema.json which defines the enum as open/resolved/unresolved. Color strings were never in the schema; the engine was the only writer out of compliance.

**`other` kind added to Interpret (2026-06-25):** Real session 0e62ec9a had `reported: "hi"` ‚Äî the operator's greeting was classified as `symptom` and poisoned the chief complaint field. The actual symptom landed in `sanity_notes`. Fix: tighten `symptom` to require a described machine problem; greetings/filler ‚Üí `other`. Engine handles `other` at session start with an INTAKE re-prompt and mid-flow with a per-stage re-ask (no state advance).

**Vault write path added (2026-06-26):** The flow previously wrote only to Storage (`/storage/v1/object/mary-memory/runtime/...`). There was no vault row writer. Added N16a `vault_fields` (code node, parses `engine.out_case` JSON string and outputs `v_status`/`v_fault_type`/`v_resolution` as flat strings) ‚Üí N16 `vault_write` (HTTP POST to `/rest/v1/vault`). Dify's template engine cannot dot-path into a JSON blob variable (e.g. `{{#engine.out_case.status#}}` doesn't resolve) ‚Äî hence the intermediate code node. Upsert strategy: `Prefer: resolution=merge-duplicates` on the `conversation_id` PK, so each turn updates the row in place and the DB-default `created_at` is preserved. Uses `SUPABASE_SERVICE_KEY` (service-role, bypasses RLS) ‚Äî same credential as N15. `v_resolution` fallback chain: `resolution_note` (CLOSEOUT path) ‚Üí reconstructed `"resolved via {procedure}; cause {confirmed_cause}"` (Mary-led RESOLVED, confirmed by inspecting real session c5abedce) ‚Üí `""`. `resolution_note` is absent on Mary-led closes; only `procedure` + `confirmed_cause` are written to runtime.json in that path.

**`__none_of_these__` clarify escape (2026-06-26):** DECLINE-from-CLARIFY was unreachable. The CLARIFY `else` branch (non-matching `check_answer` ‚Üí burn ‚Üí decline) only fired for `check_answer`-classified input; a real rejection classified as `other` (CLARIFY re-ask, no burn) or `symptom` (‚Üí SANITY, no burn), so `CLARIFY_CAP` never tripped. Fix: append a fixed `NONE_OF_THESE` (`"__none_of_these__"`) option to every clarify MCQ ‚Äî done once in `_await` (the single render choke point, so all 3 clarify-emit paths inherit it), NOT in `case._clarify` and NOT in the LLM prompt. Recognize its selection at the TOP of the CLARIFY stage, before the `other`/`symptom` handlers can swallow it: burn a slot, DECLINE at cap (terminal, unresolved), else re-ask + stay open so a real symptom next turn re-matches. Why deterministic value not display text: the token must survive Interpret's classification ‚Äî the label "None of these‚Ä¶" would itself get tagged `other`/`symptom` and re-trigger the deadlock. Frontend renders a friendly label, sends the raw token. Scoped to CLARIFY only ‚Äî mid-flow `other` re-ask-without-burning (the 2026-06-25 safety fix) is untouched. Separate from the v5.2 "clarify floor" item.

**Sanity detail enriches `reported` ‚Äî Option A (2026-06-29):** The SANITY beat used to append the operator's whole answer to `sanity_notes` only, so symptom detail volunteered when prompted ("finish getting worse, coolant smells off") never reached `reported` ‚Äî MATCH ran on a thin chief complaint. Fix: in the SANITY branch, also append the answer to `reported` **verbatim** (exact words, no paraphrase) unless it's a bare no-change reply (`no`/`nope`/`nothing`/`nothing changed`/‚Ä¶). `sanity_notes` still keeps the full answer ‚Äî append-to-`reported`, NOT move. Strict subset of a future B design: when a `change_note` Interpret class lands, tighten what flows into `reported` to exclude change-antecedents (`new tool`), which stay recoverable in `sanity_notes`. MATCH reads `case["reported"]` (the accumulation), never `sys.query` ‚Äî that's what lets the score climb as detail accrues. Falls out for free: an alarm code typed in the sanity turn now reaches the deterministic regex (no separate widening needed), and richer DECLINE capture.

**ELICIT loop ‚Äî probe vague low-score descriptions (2026-06-29):** A clear-but-low MATCH (`top < DECLINE_FLOOR`, not bunched) used to DECLINE immediately, refusing a vague description of a *known* fault. Now it PROBES. New ELICIT stage mirrors SANITY: elicit answers fold into `reported` (the Option-A path) ‚Üí MATCH re-scores the accumulated whole each turn. Kept DISTINCT from CLARIFY by design ‚Äî ELICIT is open free-text for all-low/vague (probe for signal); CLARIFY is the MCQ for bunched-plausible (pick between knowns); `bunched` is still tested first so its priority is unchanged. Render: ELICIT ‚Üí `free_text` modality, no menu; round 1 broad, round 2+ targets the leading candidate's discriminators without asserting a cause. **Stop is behavioral, not score-based** ‚Äî scores are too noisy round-to-round, so the original `top <= _elicit_top` no-gain check was removed entirely. Instead track `_elicit_empty`: an answer that adds no new info (negation/filler in the broader `_ELICIT_EMPTY` set, or empty-after-normalize) increments it; real substance resets it to 0. DECLINE on `_elicit_empty >= 2` (operator tapped out, reason `elicit_tapped_out`) OR `elicit_attempts >= ELICIT_CAP=3` (backstop, reason `elicit_exhausted`). The score now does exactly one thing: clear the floor ‚Üí MATCH.

**`elicit_attempts` empty-output bug (2026-06-29):** First import of the ELICIT yml returned empty engine output in Dify. Cause was NOT the code (it returns a valid move for the same input) ‚Äî it was missing DSL plumbing. A threaded conversation counter needs 6 sites; only the `code` had been edited. The engine node returned `out_elicit_attempts` with no matching `outputs` declaration ‚Üí Dify rejected the node. Fix: mirror `clarify_attempts` across conversation_variables / node input binding / node outputs decl / variable-assigner write-back (+ the param and `_out` key already in code). See the durable rule in CLAUDE.md.

## How things work

**Engine advance principle:** the engine only advances state on positive evidence (`check_answer`, `confirmation`, `closeout_info`). Any unclassifiable input (`other`) holds position and re-asks. This is why `other` mid-flow is not converted to `confirmation` ‚Äî absence of signal ‚âÝ positive confirmation.

**`reask=True` payload flag:** when the engine re-emits a move due to `other`, it sets `reask=True` in the payload. `_brief` checks for this and prepends "Didn't catch that ‚Äî could you clarify?" to the Render brief, giving the LLM the right framing without duplicating re-ask logic per move type.
