# distill ‚Äî notes (rationale & how-things-work)

## Decisions

**review.html dropped (2026-06-25):** The browser-based review gate was replaced with stdout dry-run + `--apply`. Rationale: the terminal diff (narrative + before‚Üíafter changed fields only) is faster to read than opening a browser, and the two-step flow (read ‚Üí re-run with --apply) is a simpler approval path than a download button. Decision log is still written on --apply for audit trail.

**ADD candidates deferred (2026-06-25):** The knowledge schema (`contracts/schema.json` `$defs.failure_mode`) requires 7 fields: `id, component_ref, title, guardrail, case_count, symptom_signature, causes`. A runtime session can only supply `id` (derived from symptom). Everything else needs human authoring. Deferred ADDs are listed in stdout with the missing field names so the author knows exactly what to fill in.

**storage.py added (2026-06-26):** Single I/O module for the distill block. `StorageClient(source="bucket")` uses `SUPABASE_SERVICE_KEY` (falls back to `SUPABASE_API_KEY`) from `.env`. `source="local"` is file-system only (no network) ‚Äî used by smoke tests so they don't need Supabase credentials. `list_promotable` queries vault with `status=eq.resolved&promoted_at=is.null`; `mark_promoted` PATCHes vault with `promoted_at=now()` and `promoted_into_version`. Dify flow uses `SUPABASE_SERVICE_KEY` as the env var name; local `.env` uses `SUPABASE_API_KEY` ‚Äî storage.py tries both.

**promote.py bucket integration (2026-06-26):** `build_change(fp, rt, ...)` refactored to `build_change(conv_id, rt, ...)` ‚Äî no longer derives conv_id from filename. Bucket mode adds `--machine-id` and `--source` flags; absence of `--knowledge`/`--runtime` triggers bucket path. `--apply` in bucket mode calls `storage.put_knowledge` (upserts to Storage) and `storage.mark_promoted` for active (non-deferred) changes only. First real promotion: DMC80FD-01 `2026-06-26.1` ‚Äî 2 ENRICHs on `fm_coolant_concentration_low` (conv_ids `c5abedce`, `949705bb`).

**draft_add.py added (2026-06-26):** LLM-assisted drafting for ADD failure_modes. Pure function `draft_add(runtime, knowledge)` ‚Äî no file I/O, no Supabase ‚Äî so it can move server-side later. Prompt bakes in two non-negotiable rules: (1) `operator_phrases` verbatim from `reported`/`resolution_note` ‚Äî this is what makes the FM reachable by the next operator who uses the same words; (2) `component_ref`, `title`, `guardrail` emitted as `"TODO: human must author ‚Äî do not fabricate"` ‚Äî these require machine knowledge + naming conventions the LLM doesn't have. `id` and `provenance` are stamped by Python after the LLM call, never trusted from LLM output. CLI writes `out/add_<conv>.draft.json` on success, `out/add_<conv>.draft.RAW.txt` + exit 1 on parse failure after retry. OPENAI_API_KEY lives in Dify, not the local `.env` ‚Äî must be added separately for local runs.

**ADD path wired end-to-end in promote.py (2026-06-26):** `build_change` ADD branch no longer emits a skeleton ‚Äî it returns `after:None, runtime:rt, deferred:True` (carries the runtime so dry-run can draft it; `deferred:True` keeps `apply_changes` from ever appending an un-authored body). Dry-run calls `write_add_draft` ‚Üí `draft_add` ‚Üí `out/add_<conv>.draft.json` (LLM-drafted, TODO sentinels). `--apply` calls `load_completed_draft(path, schema)`: rejects if file missing, if `"TODO: human must author"` substring survives, or if `validate_fm` (wraps the FM in `{"failure_modes":[fm]}` so the `$ref` resolves) finds errors ‚Üí prints `ADD blocked`, leaves `deferred:True` so it's skipped. On a clean draft it stamps `provenance:[conv]`/`evidence_count:1`, flips `deferred:False`, sets `target_id` to the draft's `id` ‚Äî then the existing `apply_changes` appends it like any non-deferred change (so ENRICH and ADD share one merge/version-bump/log path). `active`/`deferred` are recomputed inside the `--apply` block *after* the ADD gate (they used to be module-level before it). The per-draft gate resolves a default schema (`a.schema or ../contracts/schema.json`) so it's real even without `--schema`; the whole-knowledge `validate_schema` stays gated on explicit `--schema` so the smoke test (whose BASE_KB fixture is intentionally non-conformant) still runs. **Publish is FM-agnostic:** bucket `--apply` calls `store.put_knowledge(machine, candidate)` on the whole object ‚Äî no op branching ‚Äî so an ADD-merged knowledge.json publishes identically to an ENRICH one (verified by round-trip through a local StorageClient). Idempotency for ADD: `--apply` skips a conv already in any FM's provenance (bucket mode also filters via `promoted_at IS NULL`).

## How things work

**promote.py flow:** load knowledge ‚Üí for each resolved runtime, classify (enrich/revise/add) ‚Üí idempotency check (skip if conv_id already in provenance) ‚Üí print per-change block (narrative + diff) ‚Üí on --apply: apply non-deferred changes, bump _meta version, schema-validate (if --schema given), write knowledge-{machine}.json + decision-log + candidate.json/changes.json.

**Idempotency:** conv_id = filename (since `id` is null in real files). Already-in-provenance sessions are counted as "already promoted" and skipped silently.

**evidence_count:** always derived as `len(provenance)`, never incremented independently.
